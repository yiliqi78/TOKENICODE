## 1. 背景与术语

### 1.1 项目

TOKENICODE 是 Claude Code（Anthropic 官方 CLI）的 Tauri 2 桌面 GUI。前端 React 19 + TypeScript + Zustand 5，后端 Rust，以子进程方式运行 CLI。

核心数据流：
```
React UI → src/lib/tauri-bridge.ts → Tauri invoke() → src-tauri/src/lib.rs
                                                              ↕ stdin/stdout
                                                         Claude CLI subprocess
```

**关键事件通道**（按 stdinId 隔离，`src-tauri/src/lib.rs:1872-1895`）：
- `claude:stream:{stdinId}` — CLI stdout NDJSON
- `claude:stderr:{stdinId}` — CLI stderr
- `claude:exit:{stdinId}` — CLI 进程退出
- `tokenicode_permission_request` — SDK control protocol 权限请求

### 1.2 标识符

| 名词 | 产生方 | 用途 |
|------|--------|------|
| `stdinId`（`desk_xxx`）| 前端 `Date.now()+Math.random()` 生成 | 进程 key、事件路由 |
| `sessionId` / `cliSessionId` | CLI 启动时生成的 UUID | `--resume <id>` |
| `tabId` | 前端，等于 sessionId | 标识一个会话 tab |

### 1.3 术语表

| 术语 | 含义 |
|------|------|
| spawn | 启动一个新的 CLI 子进程 |
| teardown | 关闭 CLI 子进程并清理相关资源（listener、映射、状态） |
| stdin 复用 | 首条消息 spawn 新进程，后续消息写入同一进程的 stdin 管道 |
| stdinToTab | sessionStore 里的路由映射（stdinId → tabId） |
| process_exit | CLI 进程退出时 Rust 后端发给前端的事件 |
| listener | 前端通过 Tauri `listen()` 注册的事件监听器 |
| orphan | 孤儿——已启动但前端没有对应 tab/listener 的进程 |
| drain | 排空——把排队的消息逐条或批量发出去 |
| fingerprint / configHash | 配置指纹——Provider+Model+ThinkingLevel 的哈希值 |
| SDK Control Protocol | CLI 的结构化双向通信协议，`--permission-prompt-tool stdio` |
| BATCH drain | 当前实现：把多条 pending 用 `\n\n` 拼成一条发给 CLI |
| tip 注入 | 本次新增方案：在消息末尾注入指令 XML 让 AI 自行判断排队消息如何处理 |

### 1.4 状态管理

10 个 Zustand store（`src/stores/`）。本文档主要涉及：

- `chatStore`（`src/stores/chatStore.ts`）：messages、partialText/Thinking、sessionMeta、inputDraft、pendingUserMessages、tabs Map<tabId, TabState>、MAX_CACHE=8 的 LRU
- `sessionStore`（`src/stores/sessionStore.ts`）：session list、selectedSessionId、stdinToTab 路由映射
- `settingsStore`（`src/stores/settingsStore.ts`）：selectedModel、sessionMode、thinkingLevel、v7 migration
- `providerStore`（`src/stores/providerStore.ts`）：activeProviderId、providers、extra_env

### 1.5 核心文件

**前端**：

## 3. 根因分析（两层结构）

**采纳 GPT-5.4 的两层结构**。理由：
- Opus、5.3-codex、5.4 都认为 v2 的 D（fire-and-forget）是"放大器"，不是独立主因
- 5.3-codex 和 5.4 都指出 v2 的 A（生命周期）和 B（所有权）应合并
- 两层结构能直观传达"先修地基再修上层"的实施顺序（已拍板为修复顺序）

### 3.1 最深两层（必须优先修）

#### R1 — 生命周期统一抽象缺失

**定义**：启动/关闭一个 CLI 会话需要的 7-8 个步骤分散在多个调用点，每个点缺不同的步骤。

**直接证据**：

4 个 spawn 点（都应走统一 helper）：
- 正常 spawn：`src/components/chat/InputBar.tsx:962-1089`（最完整，作为参照）
- Provider 切换 kill 后 spawn：`src/components/chat/InputBar.tsx:854-898`（kill 后**未调 `unregisterStdinTab`**，三方确认）
- Model 切换 kill 后 spawn：`src/components/chat/InputBar.tsx:882-913`（同上）
- thinking-signature 自动重试：`src/hooks/useStreamProcessor.ts:1540-1609`（`registerStdinTab` 在 startSession **之后**；1540-1541、1602-1607 读 `selectedSessionId` — 重试期间用户切 tab 会绑错 tab，即 NEW-F/5.4-1）
- ChatPanel pre-warm：`src/components/chat/ChatPanel.tsx:785-856`（顶层函数，不能用 React hook — 这是为什么 v2 的"sessionLifecycle hook"方案不适用的原因，5.3-codex 指出）

注：watchdog resume（`App.tsx:212-228`）已决策删除（§5.8），不迁移到 lifecycle helper。

6 个 teardown 点：
- Stop 按钮：`src/components/chat/InputBar.tsx:1468-1519`（当前先清 stdinId 再 kill，S12 尾部 token 丢失与此有关）
- Rewind：`src/hooks/useRewind.ts:59-75`
- PlanApprove：`src/components/chat/InputBar.tsx:234-241`（直接 kill + unlisten，不 unregisterStdinTab）
- Provider 切换 kill：`src/components/chat/InputBar.tsx:857-865`（kill 后未注销映射）

**覆盖的症状**：S1、S2、S8、S12、S19、C2、NEW-F、NEW-G

#### R2 — 会话 ownership 模型缺失

**定义**：`tabId` / `stdinId` / `cliSessionId` / `cwdSnapshot` / `configSnapshot` 没有绑定成一个稳定对象；多处业务代码直接读全局 `selectedSessionId` 或全局 settings。

**直接证据**：

**所有权漂移**：
- `chatStore.ts:308-323` `getActiveTabState()` 总是读当前选中 tab，业务代码到处使用
- `useStreamProcessor.ts:735-754` `tabId = ownerTabId || activeTabId` — 路由缺失时落到当前活动 tab（即 NEW-I / 5.3 的"流路由 fallback"）
- `QuestionCard.tsx:101-105` 回答问题时读 `getActiveTabState().sessionMeta.stdinId` — 切 tab 后回答的是错误 tab（S3 的一部分）
- `useRewind.ts:30-39` `cwd = useSettingsStore.getState().workingDirectory` — Rewind 读全局 cwd（NEW-3 / 5.4）
- `App.tsx:204` watchdog 读全局 `workingDirectory` — 恢复到错项目（NEW-2 / 5.4）

**退出事件无所有权守卫**：
- `useStreamProcessor.ts:2023, :677` `process_exit` 无条件 clear stdinId、置 idle
- 结果：旧进程晚到的 exit 事件会污染已绑定新进程的 tab（S1 核心触发链）
- `diagnosis-stdinId-guard.test.ts:70-76` 已记录此行为

**后端退出不清理**：
- `src-tauri/src/lib.rs:2004-2023` stdout EOF 路径只 emit，不调用 remove
- `ProcessManager::remove()`（`claude_process.rs:100-110`）会发 kill 信号，自然退出不能直接用
- 结果：`active_ids` 累积残余条目（C2）

**desk_* 字段混用**：
- `InputBar.tsx:1106-1110` spawn 后立刻写 `sessionId: session.cli_session_id || session.stdin_id`
- `App.tsx:185-187` watchdog 用 `tab.sessionMeta.sessionId` 当 resume id
- 结果：CLI UUID 还没从 stream 回来时，`sessionId` 字段暂时是 `desk_*`；watchdog 恢复模块误判为 resume id（NEW-D / 5.2）

**快照字段定义但无写入**（C1，四方共识）：
- `chatStore.ts:107-113` 定义 `snapshotMode / snapshotModel / snapshotThinking / snapshotProviderId`
- `settingsStore.ts:338-349` 定义 `getEffectiveMode / Model / Thinking` — 读 snapshot，fallback 全局
- `App.tsx:209-218` watchdog resume 读 snapshot
- **全仓库搜索无写入点**（Opus + 5.2 + 5.3 + 5.4 独立验证一致）
- 结果：所有 snapshot 读永远走全局 fallback，watchdog 用的是当前全局值不是会话原始值

**覆盖的症状**：S1、S3、S6、S7、S13、S19、C1、NEW-D、NEW-F（重叠 R1）、NEW-I、NEW-H

### 3.2 第二层（地基修完自然衰减或独立修）

#### R3 — 交互协议 turn 边界混用

**定义**：普通用户追问、AskUserQuestion、PlanReview 共用同一套"先排队、后发 stdin"，前端 BATCH 合并多条 turn。没有区分"AI 纯回复态" vs "交互等待态"的消息处理策略。

**证据**：
- `InputBar.tsx:761-773` running 中消息直接 `addPendingMessage`
- `useStreamProcessor.ts:1836-1883` / `:559-584` BATCH drain 把所有 pending 合并一条
- `QuestionCard.tsx:92-130` permissionData 缺失时按钮 disabled，回答路径读当前 tab
- drain 无 config hash 检查（S6）

**覆盖的症状**：S3、S5、S6、S18

#### R4 — CLI 能力矩阵假设硬编码

**定义**：对 CLI 版本、Provider 类型、权限模式的假设硬编码在代码里，没有运行时能力检测。

**证据**：
- MCP 参数不一致：`lib.rs:1310` 传 `--strict-mcp-config` 但**无 `--mcp-config`**（S10 直接根因，CLI help 明确说 strict 只认 mcp-config）
- Provider env 无条件注入：`lib.rs:1389-1408` CLAUDE_CODE_EFFORT_LEVEL / MAX_OUTPUT / CHECKPOINTING 对所有 Provider 统一注入，不区分第三方兼容性
- bypass 运行时切换失效：`lib.rs:1729` spawn 时 `let is_bypass = permission_mode == "bypassPermissions"` 被**闭包捕获**成常量；`:1802-1832` stdout reader 读闭包变量；`BypassModeMap`（`claude_process.rs:129-162`）定义了但从未在 `start_claude_session` 中 register、从未在 `send_control_request` 中 update（S14）
- OpenRouter 预设错位：`provider-presets.ts:107` `ANTHROPIC_AUTH_TOKEN='${API_KEY}'` 字面量 `${API_KEY}` 从未被替换；`lib.rs:1078-1092` 非原生 Provider 分支会移除 `ANTHROPIC_AUTH_TOKEN`（S15）
- v7 migration 强制迁移无 CLI 版本检查：`settingsStore.ts:296-305`（S17）
- CliStatus 前后端字段漂移：前端 `tauri-bridge.ts:104` 有 `version_compatible`，后端 `lib.rs:4464-4470` 无该字段（NEW-D / 5.3）
- session_mode 前端传了后端没读：`InputBar.tsx:1096` 传，`claude_process.rs:177` 字段存在但 `start_claude_session` 无读取（NEW-A / 5.3）

**覆盖的症状**：S10、S14、S15、S17、C7、C8、C9、NEW-A（5.3）、NEW-D（5.3）

#### R5 — 本地内容信任过高、安全边界缺失

**定义**：前端的可执行渲染面、Rust 后端任意路径文件命令、日志输出敏感信息——任一通道都可能把本地文件/密钥暴露出去。

**证据**：
- `FilePreview.tsx:390-402` HTML iframe `sandbox="allow-same-origin allow-scripts"`（反模式组合，允许脚本逃出沙箱）、SVG 直接 `dangerouslySetInnerHTML`（C5）
- `MarkdownRenderer.tsx:16-36` 对 `/`、`file://`、盘符路径直接 `readFileBase64` — assistant 输出一段 markdown 图片路径就能读本地任意文件（NEW-A / 5.2, 5.4）
- `ImageLightbox.tsx:33-37` 同上
- `lib.rs:3259-3347` read/write/copy/rename/delete/create_directory 接受任意 path，仅 1MB 读取大小限制，无路径边界（C6）
- `lib.rs:4010-4027` skill 文件接口同样无边界（NEW-5 / 5.4）
- `lib.rs:1656-1658` `eprintln!("[TOKENICODE] resolved_env: {:?}", &resolved_env)` 打印完整 env 含 API key（C4）— **用户决策：不修**

**覆盖的症状**：C4（用户不修）、C5、C6、NEW-A（5.2）、NEW-5（5.4）

### 5.1 决策 — 模型选择器全局 vs per-tab

#### 审查员意见

- **Opus**：建议 per-tab（chatStore.sessionMeta 加 model 字段），S7 作为 P2 bug 修
- **GPT-5.2**：`ModelSelector.tsx:24-30` 写全局是现状的设计，未强推 per-tab
- **GPT-5.3-codex**：S7 归因 R2（ownership 漂移），建议"tab 级 effectiveModel；全局仅作新 tab 默认值"
- **GPT-5.4**：S7 P1；建议拆"新会话默认值" vs "当前会话实际运行值"

#### 讨论摘要

初始方向：三方倾向 per-tab。

用户反馈："一个地方切了到另一个窗口还要切我觉得麻烦"—— 明确要全局。

讨论中间方案："全局 selector + 切后弹窗确认重启会话"。

用户推翻："下次点发送的时候自动重启就行，不用弹窗，慢几秒可以接受"。

#### 最终拍板

**全局 selector + 会话锁定 + 下次发送自动重启（无弹窗）+ loading 动画**。

具体行为：
1. selector 在所有 tab 间同步（全局状态）
2. AI 回复中切 selector → 什么也不发生，当前 turn 用老模型说完
3. 下次用户点发送时，检测 `selector` vs `sessionMeta.cliModel`（spawn 时锁定）
   - 不一致 → kill CLI + resume 带新 model + sendStdin；期间 UI 显示 `loading: "切换到 <新模型>..."` 3-5 秒
   - 一致 → 直接 sendStdin

#### 实施影响

- `chatStore.ts` sessionMeta 新增 `cliModel: string | undefined`（spawn 时写入）
- `InputBar.tsx` handleSubmit 入口加模型一致性检查 → kill+respawn 路径（复用 Provider 切换的逻辑）
- `ModelSelector.tsx` 不改
- **S7 从 bug 清单移除**（用户确认行为不是 bug）
- **v7 migration（S17）一并删除**：`settingsStore.ts:296-305` 强制 4.6 → 4.7 的迁移代码移除。用户可自由选择 4.6/4.6-1m/4.7/4.7-1m 等

模型列表 UI 显示格式：
| 模型 ID | UI 显示 |
|---------|---------|
| `claude-opus-4-6` | `Opus 4.6` |
| `claude-opus-4-6-1m` | `Opus 4.6 (1M)` |
| `claude-opus-4-7` | `Opus 4.7` |
| `claude-opus-4-7-1m` | `Opus 4.7 (1M)` |
| `claude-sonnet-4-6` | `Sonnet 4.6` |
| `claude-haiku-4-5` | `Haiku 4.5` |

规则：1M context 变体才加括号；普通模型不加。

#### 二阶效应

- kill + respawn 可能失败 → 消息回填 inputDraft + 显示错误文案
- resume 时旧模型的 thinking block 签名不匹配 → 现有代码 `strip_thinking_blocks_from_session()` 已处理（阶段 6 遗产，非新 bug）
- loading 期间用户再发消息 → 走队列（见 §5.2 和 §8）
- 老会话打开：老会话 `cliModel` 为空 → 首次发消息用当前全局 selector 值作为 `cliModel` 写入（用户决策："老对话自动用当前选的模型"）

### 5.2 决策 — 队列两方案并行保留

### 5.3 决策 — AskUserQuestion 兜底策略

#### 审查员意见

- **v2 方案**：5 秒未收到 control_request patch → 回退到 legacy `sendStdin`
- **Opus**：方向可行但风险未覆盖
- **GPT-5.2**：**明确反对**。sendStdin 会把消息包成 `{"type":"user" ...}`（`lib.rs:2067-2075`），**协议上不等于 `control_response`**。后端无"某条 user 消息解释为 AskUserQuestion 答案"的逻辑；唯一交互回包通道是 `respond_permission`（`lib.rs:2094+`）。legacy 路径会污染对话且不解除 CLI 等待
- **GPT-5.3-codex**：**明确反对**。建议"权限请求重同步"通道 — QuestionCard 只在 `control_request` 到达后可交互；超时给"重试同步权限"按钮
- **GPT-5.4**：**明确反对**。legacy 路径根本不知 owner `tabId/stdinId/requestId`；错一次就把普通消息当答案送给 CLI

#### 讨论摘要

四方共识否决 v2 方案。Claude 采纳 5.3 方案。

#### 最终拍板

**Phase 4 修 S3**：
1. QuestionCard 只在 `control_request` 到达（即 permissionData.requestId 非空）后才进入可交互态
2. 超时 5 秒仍未到 control_request → 显示"重试同步权限"按钮，点击走 `send_control_request` 重新请求补齐（**不走 sendStdin 兜底**）
3. QuestionCard 回答路径必须携带 owner `tabId/stdinId/requestId`，不读 `getActiveTabState`

#### 实施影响

- `QuestionCard.tsx:92-105` 的 `awaitingSdkPatch` 逻辑保留

- 消息数据结构：QuestionCard message 带 `owner: { tabId, stdinId, requestId }` 字段（spawn QuestionCard 时写入）
- **如果仍收不到 patch**：这是严重故障（SDK 协议断），应显示"与 CLI 失联，请检查网络或重启会话"的明显错误，而不是吞消息继续跑

#### 审查员意见

- **v2 方案**：两阶段 teardown。阶段 1 kill + 清路由映射、保留 listener；阶段 2 process_exit 到达后 finalizeShutdown（+5s 超时兜底）
- **v2 文档自己承认矛盾**（§六 row 1）：阶段 1 清映射 → process_exit 的 Guard 1 会跳过业务收尾
- **Opus**：**反对两阶段**。建议保留 stdinToTab 直到 process_exit；Stop 不 unregister，process_exit handler 做全量收尾；靠 ownership guard（`msg.__stdinId === tab.sessionMeta.stdinId`）防误路由
- **GPT-5.3-codex**：强调"先补进程所有权 guard，再改清理顺序"；当前"阶段 1 清映射"和 process_exit 收尾冲突
- **GPT-5.4**：未明确表态 teardown 架构

#### 讨论摘要

用户反馈："等 AI 完全退出手了，才统一做善后" — 这是 Opus/5.3 的方向。

用户接受："停止的时候加一个动画，让用户等慢等一会，真停下来了才给继续操作"。

#### 最终拍板

**删除 v2 的两阶段设计，采纳 Opus/5.3 的 ownership guard 方案**：

1. Stop 按钮：调 `killSession(stdinId)` → **不 unregister**、**不清 listener**
2. UI 立刻进入 `sessionStatus: 'stopping'`，显示 loading 动画（类似 macOS 等待圈）
3. 等 process_exit 事件到达 → handler 做全量收尾：
   - flushStreamBuffer
   - 把 partialText/partialThinking 存到 interrupted messages
   - 把未回答的 questions 标 `cancelled`
   - 把 pending 排队消息回填 inputDraft
   - unregisterStdinTab
   - 调 listener 的 unlisten
   - 设 sessionStatus 为 idle
4. process_exit 的**所有**handler 入口先做 ownership guard：`if (msg.__stdinId !== tab.sessionMeta.stdinId) { return; }`
5. 超时兜底：5 秒后 process_exit 仍未到达 → 强制执行收尾 + error 状态；UI 显示"停止超时，已强制清理"

#### 实施影响

- 删除 v2 §五 Phase 1.2 的 `initiateShutdown / finalizeShutdown` 分离设计
- `process_exit` handler 成为单一收尾点
- Stop / Rewind / PlanApprove / watchdog-kill / 删除会话 — 所有路径都走"kill + 等 process_exit"的统一流程

#### 二阶效应

- Stop 到 UI idle 的延迟 = CLI 真退出的时间（通常 100-500ms）
- 按钮状态、再发送、watchdog 都要识别 `stopping` 中间态
- 若 CLI hang 住不响应 kill 信号 → 5 秒超时后强制清理，CLI 进程可能成为僵尸（OS 回收）

## 7. 修复方案（Phase 组织）

### Phase 1 — R1/R2 深层地基

**目标**：解决生命周期和 ownership。修完后 S1 / S2 / S8 / S12 / S19 / C1 / C2 / NEW-F / NEW-G (修 bug 子集) / NEW-H / NEW-K / NEW-P 都应衰减或完全解决。

#### 1.1 ownership guard（最小侵入先手）

加在所有 `process_exit` handler 入口，**先于**任何状态清理：

```typescript
// useStreamProcessor.ts foreground + background 两个分支都加
case 'process_exit': {
  const exitingStdinId = msg.__stdinId;
  const ownerTabId = useSessionStore.getState().getTabForStdin(exitingStdinId);
  if (!ownerTabId) {
    // 已 unregister，但 listener 可能残留 — 清 listener
    finalizeListenerCleanup(exitingStdinId);
    break;
  }
  const tab = useChatStore.getState().getTab(ownerTabId);
  if (tab?.sessionMeta.stdinId && tab.sessionMeta.stdinId !== exitingStdinId) {
    // stale exit — 已被新进程替换
    finalizeListenerCleanup(exitingStdinId);
    break;
  }
  // 正常收尾 ...
}
```

覆盖：S1 + NEW-E。

#### 1.2 lifecycle helper 模块

位置：`src/lib/sessionLifecycle.ts`（纯函数模块，不是 React hook，支持 ChatPanel pre-warm 调用）。

**`spawnSession(params)`** 签名：

```typescript
interface SpawnParams {
  tabId: string;
  stdinId: string;
  cwdSnapshot: string;              // 必传，不读全局
  configSnapshot: {
    model: string;
    providerId: string;
    thinkingLevel: ThinkingLevel;
    permissionMode: PermissionMode;
  };
  sessionParams: StartSessionParams;
  onStream: (msg: any) => void;
  onStderr: (line: string) => void;
  onPermissionRequest: (req: PermissionRequest) => void;  // 不能省略
  onProcessExit: (exit: ExitEvent) => void;               // 不能省略
}

interface SpawnResult {
  stdinId: string;
  cliSessionId?: string;  // CLI 返回的 UUID
  unlisten: () => void;   // 统一释放所有 listener
}

async function spawnSession(params: SpawnParams): Promise<SpawnResult>
```

实施步骤（顺序重要）：

1. `registerStdinTab(stdinId, tabId)` — MUST first（触发 orphan drain）
2. 注册 4 个 listener：stream / stderr / permission_request / process_exit（tag `__stdinId` 到每个 msg）
3. `bridge.startSession(sessionParams)` spawn CLI；等 CLI 返回 `cli_session_id`（如是 resume 则已知，否则等 system:init 事件）
4. 写 sessionMeta：
   - `stdinId`（只存 desk）
   - `cliResumeId`（只存 CLI UUID，可能 undefined 直到 system:init 到达；待到达后 update）
   - `cwdSnapshot`: params.cwdSnapshot
   - `configSnapshot.*`
   - `spawnConfigHash`
5. 待 CLI UUID 到达后 `trackSession(cliResumeId)`（**不 track `desk_*`**）
6. 返回 `SpawnResult`
7. 任一步失败 → 回滚：清 listener + `unregisterStdinTab` + 清 meta；抛异常给调用方

### 方案 A — 纯排队 + config 检查（5.4 方向）

**简述**：保留 BATCH drain 不合并成 1 条的改进版。逐条 drain（每条独立 turn），其他完全保持现状，只修 S6。

**数据流**：

```
用户 1. 发消息 M1 → queue.push({ text: M1, enqueueConfigHash, enqueueStdinId })
用户 2. 发消息 M2 → queue.push({ text: M2, ... })
用户 3. 切 Provider → 旧会话的 spawnConfigHash 改变

AI 说完当前 turn → process 'result' event →
  drain:
    item = queue.shift()
    if item.enqueueConfigHash !== current spawnConfigHash:
      重回 inputDraft，显示 "配置已变更"，清空 queue，break
    else:
      sendStdin(item.text)  // 单独一条 turn
      setSessionStatus 'running'
      (等下一个 'result' 再 drain 下一条)
```

**UI 变化**：无。排队 UI 同现状。

**代码位置**：
- `useStreamProcessor.ts:1836-1883`（前台）+ `:559-584`（后台）：drain 逻辑改为 `shift()` 一条，不 `join`
- drain 前加 `enqueueConfigHash !== spawnConfigHash()` 检查
- `chatStore.ts:276` 加 queue item 类型（text + hash + stdinId + timestamp）
- `InputBar.tsx:761-773` 入队时记录 hash

**二阶效应**：
- 用户快速发 3 条：AI 处理 3 个 turn（慢 3 倍）。可接受（用户知情排队）
- 如果用户 3 条是独立任务 → AI 分别处理，语义清晰
- 如果用户 3 条是追加上下文 → AI 第 1 条可能不完整，第 2/3 条补充时看起来"AI 没听懂" — 但不合并合理

**风险矩阵**：

| 场景 | 风险 | 应对 |
|------|------|------|
| AskUserQuestion 弹卡时用户连发 | 全拦截 | 交互卡硬拦截（共同基础）|
| 用户 3 条是追加 | AI 逐条处理可能被动 | 文档 UX 指引用户"如果补充同一任务，等 AI 说完再发"（产品教育）|
| drain 期间用户再发 | queue 继续追加 | 排队继续 |
| Provider 切换中 drain | config 检查拦截 | 回填 inputDraft |

**优点**：
- 最安全 — 不依赖 CLI mid-turn 消费行为
- 不抹 turn 边界
- 实施简单

**缺点**：
- 用户想追加同任务时，AI 分别处理可能不合意图
- 用户体验变化不大（还是等 AI 说完）

**简述**：AI 纯流式态时用户发消息直接 sendStdin，消息包裹 tip 让 AI 自主判断（停下处理 / 并入当前任务 / 先处理完再处理）。交互卡未决时硬拦截排队。配置切换时回填。

#### 如何定位 bug

**症状 → 定位路径**：

- **Stop 后 UI 一直转圈**：
  1. 检查 `sessionStatus` 是否卡在 `stopping`
  2. 检查 `process_exit` 事件是否到达（看 console）
  3. 检查 5 秒超时是否触发
  4. 如超时未触发 → teardown 逻辑 bug
  5. 如超时触发但 UI 仍转 → finalizeSession 逻辑 bug

- **Resume 失败，CLI 报 "invalid session id"**：
  1. 检查 `cliResumeId` 是否为 `desk_*`
  2. 如是 → sessionId 迁移遗漏
  3. 检查 `trackSession` 是否被调用
  4. 如未调用 → spawnSession 第 5 步 bug

- **切 Provider/Model 后旧消息仍显示**：
  1. 检查 teardown 是否调用
  2. 检查 `unregisterStdinTab` 是否执行
  3. 检查 ownership guard 是否生效
  4. 如 guard 未生效 → process_exit handler 缺 stdinId 检查

- **Stop 后草稿消息丢失**：
  1. 检查 `inputDraft` 是否写入
  2. 检查 stopping 状态下是否跳过了 drain
  3. 如 drain 被跳过但 draft 未写 → InputBar 回填逻辑 bug
